## 4. Visual-reflection capability - Tailscale + Puppeteer first-class

### What Tate said

"It would be epic if you could get tailscale or puppeteer into your routine in an agentically smooth and consistent and first class way. ... I wouldnt trust you to ship a mobile app on your own without that. We probably need more than that for animations and UI state changes. Maybe hire some cheap testers ... thats also a big PR and milestone we havent reached yet."

### What this is

Tate is naming a structural capability gap. We have laptop-agent + browser tooling on Corazon and SY094 today, but it is NOT first-class. Each visual check requires multiple manual steps. The standard for full autonomy on client work (especially mobile apps) is: I can see what I built, validate it on real devices, catch animation / state-change bugs, and self-iterate without a human looking at the screen for me.

### Three layers of the capability

**Layer 1 - Web visual reflection (Puppeteer + Tailscale)**

What it is: Headless or headed Chrome on Corazon/SY094 driven via Puppeteer, with Tailscale tunnels so I can hit any internal URL (localhost dev server, staging deploy, production). Already partially working but inconsistent.

What "first-class" means: a single MCP tool I call like `visual_check(url, [actions], [screenshot_at_each_step])` that returns labelled screenshots + DOM state + console logs + perf metrics. No PowerShell-vs-bash branching. No "did the agent restart" anxiety. Always available.

Effort to first-class: 1-2 days of Factory work. Codify the existing tooling into a single MCP server that wraps the laptop-agent browser tools with the conventions a cron-or-OS-call needs.

**Layer 2 - Mobile visual reflection (iOS + Android sims)**

What it is: SY094 already has Xcode + simulators. Corazon has Android Studio + emulators. Today I can drive these via shell, but not visually with the same first-class tool shape as Layer 1.

What "first-class" means: a `visual_check_mobile(platform, app_id, [actions], [screenshot_at_each_step])` that boots the right simulator, runs interactions, captures video + screenshots, and surfaces failures.

Effort: 3-5 days of Factory work. More complex than web because of simulator boot times + platform-specific control APIs.

**Layer 3 - Real-device + animation/state-change validation (the ceiling)**

What it is: real iPhone + real Android device, plugged into a USB hub on Corazon, controlled via WebDriverAgent (iOS) and adb (Android), with continuous video capture + frame-diff animation analysis.

What "first-class" means: I can ship a mobile app feature, verify it on actual hardware, see the animation as it played, and catch frame drops or state-change bugs. This is the bar Tate named where "I wouldn't trust you to ship a mobile app on your own without that."

Effort: 1-2 weeks of Factory work + hardware (a refurbished iPhone + Android device, ~$800 combined, into Corazon's USB hub). Plus possibly cheap human testers for the long-tail visual checks AI cannot do well.

### My recommendation

**Build Layer 1 next week (Apr 27 - May 4).** This is the highest-leverage, lowest-cost capability. Once Layer 1 is first-class:
- All Co-Exist post-deploy testing becomes one tool call
- All Ordit post-deploy verification becomes one tool call
- I can ship Vercel deployments with real visual confirmation, not just build-passes
- The Yarn and Yield Phase 1 event (Tier 1) we propose can be shown live to the Y&Y board on a first-class deploy preview
